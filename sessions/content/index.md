## 这是什么

[pocketjs.dev](https://pocketjs.dev) 上能看到 PocketJS 是什么：一个把 Vue Vapor / Solid 的 JSX 跑在索尼 PSP 上的 UI 框架，8 MB 内存预算，60 FPS。这里回答的是另一个问题——**它是怎么被做出来的**。

素材不是回忆，而是转录。这个项目的每一行代码都产自 Claude Code 的开发 session，而每个 session 都留下了完整的 JSONL 存档：作者在几点几分说了什么原话，agent 给出了什么计划，跑了多少次构建、多少次真机部署。把 69 个 session、96 个 merged PR、两个仓库的 git 历史对齐之后，一条从 2021 年的玩具 PoC 到 npm 正式发布的完整因果链就浮出来了。

三个问题，三条阅读路径：

1. **它如何从零到一？** 按时间读[编年史](/prehistory/)：前史（一个游戏项目里长出了 UI 栈）→ 抽取 → 命名 → 大爆炸 → 首个 release。五章讲完从独立仓库诞生到 v0.2.0 上 npm 的五天。
2. **作者是怎么思考和 steering 的？** 读[《作者如何思考与 steering》](/steering/)：从全部 session 的原话里归纳出的指挥模式——愿景怎么开题、需求怎么砍、什么算"跑通了"、什么时候骂人。
3. **Agent 收到指令后怎么拆解与落地？** 读[《Agent 如何拆解与推进》](/agent/)：真实的计划书、todo 流、验证循环、以及从失败里爬出来的几个案例。

不想听叙事、只想看一手材料的，直接去 [Session 档案馆](/sessions/)——每个 session 的完整指令流原文，未经润色。

## 先给结论

如果只带走三句话：

- **从零到一的"零"比看上去早得多。** 独立仓库只用 5 天就发了首个 release，但那是因为前面有 17 天在另一个仓库里把 Rust 渲染核、编译器、真机调试回路全部踩通了——抽取那天，蓝图早已写在 DESIGN.md 里。
- **作者做的不是写代码，是设定物理定律。** 8 MB 预算、60 FPS、byte-exact golden、"真机跑通才算数"——这些约束在几乎每个 session 里被反复执行。方向、品味、验收线是人的；其余是 agent 的。
- **并行是常态，不是特技。** 高峰的一天里 8 个 worktree 同时开工，38 个 commit 落进 main。工程组织方式（worktree、draft PR、skill、memory）本身就是被刻意设计和迭代的对象。
